## 2. 设计目标

这份文档用于把第一版数据库设计彻底钉死，核心目标：

- 明确 MVP 所需表、字段、索引、约束和枚举
- 统一前后端、后端、测试对字段语义的理解
- 给 Go 后端建表、ORM、Repository 层实现提供直接依据
- 给后续迁移脚本、初始化脚本、测试数据脚本提供基础

第一版数据库设计遵循的原则：

- 先满足 MVP，不搞过度抽象
- 关键配置可热更新，关键状态可追踪
- 敏感数据最小化存储
- 操作审计和请求日志必须能串起来

## 3. 数据库选型建议

### 3.1 本地开发版建议

第一版建议优先使用：

- `PostgreSQL`：如果希望后续正式部署平滑升级
- `SQLite`：如果只追求本地快速开发验证

综合建议：

- 文档和代码设计按 PostgreSQL 标准来
- 若本地初期为了快，可以短期兼容 SQLite

### 3.2 命名约定

- 表名统一使用小写下划线：`snake_case`
- 主键统一使用 `id`
- 时间字段统一使用：`created_at`、`updated_at`
- 状态字段统一使用：`status`
- JSON 扩展字段统一使用：`*_json`

### 3.3 通用字段约定

所有核心业务表建议至少包含：

- `id`
- `created_at`
- `updated_at`

有状态控制的表建议包含：

- `status`

有审计意义的表可增加：

- `created_by`
- `updated_by`

12. `request_log`

说明：

- 在 `03_开发实现文档.md` 的基础上，这里把 `upstream_channel` 和 `provider_key` 关系进一步收紧为更贴近当前 MVP 的模型
- 如果你后面要更复杂的“一个 provider 下多个 channel”能力，可以在 V1.1 再拆成 `provider + upstream_channel + provider_key`
- 但第一版为了开发收口，建议先做成现在这套，别一上来把抽象搞太散

## 6. 表设计详解

## 6.1 `admin_user`

### 6.1.1 用途

后台管理员账号表。第一版虽然是单管理员，但仍按可扩展方式建表，不要把账号信息写死到配置里。

### 6.1.2 字段设计

| 字段名 | 类型 | 非空 | 默认值 | 说明 |
| --- | --- | --- | --- | --- |
| id | bigint | 是 | 自增/雪花 | 主键 |
| username | varchar(64) | 是 | - | 管理员用户名，全局唯一 |
| password_hash | varchar(255) | 是 | - | 密码哈希 |
| password_algo | varchar(32) | 是 | `bcrypt` | 哈希算法标识 |
| status | varchar(16) | 是 | `active` | 账号状态 |
| last_login_at | timestamptz | 否 | null | 最近登录时间 |
| last_login_ip | varchar(64) | 否 | null | 最近登录 IP |
| created_at | timestamptz | 是 | now() | 创建时间 |
| updated_at | timestamptz | 是 | now() | 更新时间 |

### 6.1.3 索引建议

- 唯一索引：`uk_admin_user_username(username)`
- 普通索引：`idx_admin_user_status(status)`

## 6.3 `api_client`

### 6.3.1 用途

存储你发给客户的 API Key 及其元信息。

### 6.3.2 字段设计

| 字段名 | 类型 | 非空 | 默认值 | 说明 |
| --- | --- | --- | --- | --- |
| id | bigint | 是 | 自增/雪花 | 主键 |
| name | varchar(128) | 是 | - | 客户 key 名称 |
| key_prefix | varchar(32) | 是 | - | 用于展示的前缀 |
| api_key_hash | varchar(255) | 是 | - | 客户 API Key 哈希 |
| status | varchar(16) | 是 | `active` | `active`、`disabled` |
| remark | varchar(255) | 否 | null | 备注 |
| last_used_at | timestamptz | 否 | null | 最近使用时间 |
| expires_at | timestamptz | 否 | null | 预留过期时间 |
| created_at | timestamptz | 是 | now() | 创建时间 |
| updated_at | timestamptz | 是 | now() | 更新时间 |

### 6.3.3 字段说明

- `api_key_hash` 只存哈希，不存明文
- `key_prefix` 用于后台列表快速识别，例如 `ak_live_ab12`
- 第一版不做正式配额，但字段设计可预留扩展

### 6.3.4 索引建议

- 唯一索引：`uk_api_client_key_prefix(key_prefix)`
- 普通索引：`idx_api_client_status(status)`
- 普通索引：`idx_api_client_last_used_at(last_used_at desc)`

## 8. 逻辑删除与物理删除策略

第一版建议：

- `provider`：不做物理删除，优先禁用
- `provider_key`：不做物理删除，优先停用
- `virtual_model`：不做物理删除，优先停用
- `api_client`：不做物理删除，优先停用
- `request_log`：按保留周期归档或清理
- `audit_log`：按保留周期清理，但保留时间应长于 request_log

原因很简单：

- 这是运维型系统，删太干净容易把排障线索干没了
- 第一版先求稳，别搞一堆不可恢复删除

## 11. 安全要求

### 11.1 敏感字段处理

以下字段必须加密或哈希：

- `api_client.api_key_hash`：哈希
- `provider_key.secret_encrypted`：加密
- `admin_user.password_hash`：哈希

### 11.2 不建议落库的内容

默认不应完整落库：

- 客户原始 prompt 正文
- 上游完整响应正文
- 完整 Authorization Header
- 完整上游 key

如果调试模式需要增强日志，也只允许落脱敏摘要。

## 13. 开发落地建议

### 13.1 Go 结构体建议

每张表建议至少拆成三层对象：

- `Entity`：数据库实体
- `DTO`：接口传输对象
- `BO/VO`：业务对象或返回对象

别把 ORM 结构体直接拿去对外返回，不然后面字段一变全跟着炸。

### 13.2 迁移方式建议

- 使用 SQL migration 管理建表脚本
- 不建议第一版全靠 ORM 自动建表
- 核心索引和唯一约束必须显式写进 migration

### 13.3 JSON 字段建议

JSON 字段尽量只用于：

- 参数覆盖
- 能力快照
- 日志摘要
- 系统配置

不要把核心查询字段一股脑塞 JSON，不然你后面查日志和查状态会想骂人。
