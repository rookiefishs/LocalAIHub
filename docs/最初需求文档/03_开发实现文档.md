# 03_开发实现文档

## 1. 项目概述

### 1.1 项目目标

建设一个“本地 AI 中转网关与管理后台”，对外提供统一的访问入口、统一的 API Key、统一的虚拟模型名，并在内部对接多个不同协议和多个上游服务，实现以下目标：

- 对客户端屏蔽不同 AI 厂商协议差异
- 通过虚拟模型名实现模型路由与供应商解耦
- 支持 OpenAI、Anthropic、Gemini 三类对外协议
- 支持多个上游来源，包括官方接口与第三方中转站
- 支持手动切换上游和自动熔断故障切换
- 提供模型列表接口与文本生成接口
- 统一核心生成参数，同时保留特殊参数透传能力
- 默认开启脱敏日志，支持调试日志开关
- 提供单管理员后台进行配置、监控和排障
- 第一阶段适配本地开发与单机部署，后续可平滑扩展

### 1.2 第一版设计原则

第一版建议采用“模块化单体”方案，而不是一上来拆成微服务，原因如下：

- 当前业务边界清晰，但规模尚未大到必须服务拆分
- 协议适配、路由决策、配置管理、后台管理高度耦合，单体更利于快速迭代
- 本地开发版优先保证可维护性、可调试性、可演进性
- 后续若流量增长，可按模块边界平滑拆分，不推翻现有设计

### 1.3 核心设计原则

- 分层清晰：避免把协议判断、路由决策、参数转换、日志、安全写成一坨 `if/else`
- 领域建模优先：统一抽象“渠道”“供应商”“模型映射”“路由策略”“熔断状态”
- 适配器模式：不同协议、不同上游都通过适配层接入
- 配置驱动：管理后台配置优先，代码只承载规则框架
- 安全默认开启：默认脱敏、最小权限、密钥加密存储、调试显式开关
- 面向扩展：第一版先覆盖文本生成和模型列表，后续可扩展流式输出、Embedding、图像、工具调用

## 2. 技术架构与分层

### 2.1 总体架构

系统建议分为以下几层：

1. 接入层
2. 应用服务层
3. 领域层
4. 基础设施层
5. 管理后台层

逻辑架构如下：

```text
[Client / SDK / Curl]
        |
        v
[API Gateway HTTP Layer]
        |
        +-- 对外协议入口
        |    - /v1/*
        |    - /anthropic/*
        |    - /gemini/*
        |
        v
[Application Service Layer]
        |
        +-- Auth Service
        +-- Model Service
        +-- Generation Service
        +-- Route Service
        +-- Circuit Breaker Service
        +-- Audit Service
        |
        v
[Domain Layer]
        |
        +-- Provider
        +-- Channel
        +-- Virtual Model
        +-- Route Policy
        +-- Failover Policy
        +-- RequestContext
        |
        v
[Infrastructure Layer]
        |
        +-- Repository(MySQL/SQLite/PostgreSQL)
        +-- Redis Cache / Circuit State
        +-- Upstream Adapters
        +-- Config Loader
        +-- Logger / Metrics
        +-- Crypto
        |
        v
[Admin Web UI]
```

### 2.2 分层职责说明

#### 2.2.1 Handler 层

职责：

- 接收 HTTP 请求
- 完成基础鉴权、Header 解析、协议识别
- 解析请求并绑定 DTO
- 调用 Service
- 统一响应格式和错误码转换

原则：

- 不写业务决策
- 不直接操作数据库
- 不直接拼接上游协议请求

#### 2.2.2 Service 层

职责：

- 编排业务流程
- 调用路由、熔断、适配器、仓储
- 执行统一参数转换和上下文组装
- 处理事务边界和审计记录

建议拆分：

- `AuthService`
- `ModelCatalogService`
- `GenerationService`
- `RoutingService`
- `CircuitBreakerService`
- `AdminConfigService`
- `AuditService`

#### 2.2.3 Domain 层

职责：

- 定义核心实体、值对象、领域规则
- 承载“虚拟模型 -> 路由策略 -> 可用渠道 -> 上游协议”的核心抽象

核心对象建议：

- `VirtualModel`
- `Provider`
- `UpstreamChannel`
- `RouteRule`
- `FailoverPolicy`
- `GenerateRequest`
- `GenerateResponse`
- `CircuitState`

#### 2.2.4 Repository 层

职责：

- 负责数据持久化
- 屏蔽数据库实现细节
- 提供领域友好的查询方法

原则：

- 只负责数据存取，不包含业务逻辑
- 复杂组合逻辑放在 service，不放 repository

#### 2.2.5 Adapter 层

职责：

- 负责不同上游协议和供应商的对接
- 负责协议转换、参数映射、响应归一化

建议使用接口 + 注册表模式，避免硬编码判断：

```go
type UpstreamAdapter interface {
    Protocol() string
    ProviderType() string
    ListModels(ctx context.Context, req ListModelsRequest) ([]ModelInfo, error)
    GenerateText(ctx context.Context, req UnifiedGenerateRequest) (*UnifiedGenerateResponse, error)
}
```

## 3. 服务拆分建议

### 3.1 第一版建议：模块化单体

第一版建议使用单体应用，拆成多个模块包，统一进程内通信。这样开发成本低、调试效率高、部署简单，特别适合本地开发版。

建议模块边界如下：

- `api-gateway`：对外协议入口
- `admin`：管理员后台接口
- `routing`：路由决策
- `adapter`：协议适配
- `provider`：渠道与供应商管理
- `circuitbreaker`：熔断与恢复
- `audit`：日志和审计
- `auth`：本地管理员认证、对外 API Key 校验

### 3.2 后续可拆分方向

如果后续部署规模扩大，可以沿以下边界拆分：

- 网关服务
- 路由与策略服务
- 后台配置服务
- 异步事件服务

第一版不建议拆，原因很简单：拆了之后通信复杂度、配置复杂度、调试复杂度会立刻上来，收益不大。

## 7. Redis / 缓存 / 熔断状态建议

### 7.1 Redis 角色定位

Redis 在本项目中建议承担以下职责：

- 缓存模型列表
- 缓存虚拟模型到渠道映射
- 存储熔断热状态
- 存储短期限流计数
- 存储登录会话或 token 黑名单
- 存储临时调试开关状态

### 7.2 缓存建议

#### 7.2.1 模型列表缓存

Key 设计：

```text
models:list:{protocol}:{client_id}
models:upstream:{channel_id}
```

#### 7.2.2 模型绑定缓存

Key 设计：

```text
routing:model_bind:{virtual_model}
routing:policy:{virtual_model}
```

### 7.3 熔断状态建议

熔断状态建议 Redis 热存储 + DB 持久化双写。

Redis Key 设计：

```text
cb:channel:{channel_id}
```

### 7.4 熔断策略建议

采用经典三态：

- `closed`：关闭状态，正常放行
- `open`：熔断状态，直接拒绝该渠道
- `half_open`：半开状态，允许少量探测请求

建议默认阈值：

- 连续失败 5 次进入 `open`
- 熔断窗口 60 秒
- 到时间后进入 `half_open`
- 半开连续成功 2 次恢复 `closed`
- 半开任意失败重新进入 `open`

## 8. 协议适配层设计

### 8.1 设计目标

协议适配层要解决两个问题：

1. 对外协议差异
2. 对上游协议差异

这两个层次必须拆开，不然代码会迅速烂掉。

### 8.2 两层适配结构

#### 8.2.1 对外协议适配

负责将外部请求转换为内部统一请求对象。

建议定义：

```go
type InboundProtocolAdapter interface {
    Protocol() string
    ParseGenerateRequest(r *http.Request) (*UnifiedGenerateRequest, error)
    ParseListModelsRequest(r *http.Request) (*UnifiedListModelsRequest, error)
    WriteGenerateResponse(w http.ResponseWriter, resp *UnifiedGenerateResponse) error
    WriteModelsResponse(w http.ResponseWriter, models []UnifiedModelInfo) error
}
```

#### 8.2.2 上游协议适配

负责将内部统一请求转换为具体上游请求，并把结果再转回统一响应对象。

建议定义：

```go
type OutboundUpstreamAdapter interface {
    Key() string
    Protocol() string
    Supports(provider Provider, channel UpstreamChannel) bool
    ListModels(ctx context.Context, req UpstreamListModelsRequest) ([]UnifiedModelInfo, error)
    Generate(ctx context.Context, req UpstreamGenerateRequest) (*UnifiedGenerateResponse, error)
}
```

### 8.3 统一请求模型设计

建议内部统一请求对象：

```go
type UnifiedGenerateRequest struct {
    TraceID          string
    ClientID         int64
    Protocol         string
    VirtualModel     string
    Messages         []UnifiedMessage
    Prompt           string
    Temperature      *float64
    TopP             *float64
    MaxTokens        *int
    Stop             []string
    Stream           bool
    PresencePenalty  *float64
    FrequencyPenalty *float64
    Metadata         map[string]string
    ExtraParams      map[string]any
    RawRequest       map[string]any
}
```

### 8.4 参数映射策略

建议采用“核心参数白名单 + 特殊参数透传”模式。

核心统一参数：

- model
- messages、prompt
- temperature
- top_p
- max_tokens
- stop
- stream

透传策略建议：

- 后端维护每个 adapter 的“允许透传字段白名单”
- 未列入白名单的字段默认丢弃或审计告警
- 调试模式下可记录透传命中情况，但仍做脱敏

统一响应对象建议：

type UnifiedGenerateResponse struct {

## 9. 路由决策与故障切换流程

### 9.1 路由决策核心思路

路由决策不要写死在 handler 里，而是独立为 `RoutingService`。

路由输入：

- 客户端信息
- 对外协议类型
- 虚拟模型名
- 请求参数
- 手动锁定配置
- 渠道健康状态
- 熔断状态
- 策略配置

路由输出：

- 选中的 channel
- 上游真实模型名
- 决策原因
- 候选渠道列表

### 9.2 路由优先级建议

建议整体优先级如下：

1. 校验客户端权限
2. 解析虚拟模型
3. 查询该模型绑定的可用渠道
4. 应用手动强制路由
5. 过滤手动禁用渠道
6. 过滤熔断中的渠道
7. 按优先级或策略排序
8. 选择首个可用渠道
9. 调用失败后触发自动故障切换

### 9.3 决策流程

```text
收到请求
  -> 鉴权
  -> 识别协议
  -> 解析虚拟模型
  -> 查询模型绑定渠道
  -> 应用策略规则
  -> 过滤禁用渠道
  -> 过滤熔断渠道
  -> 选择主渠道
  -> 调用上游
      -> 成功：记录成功，更新熔断状态
      -> 失败：记录失败，评估是否切换下一个渠道
  -> 返回响应
```

### 9.4 手动切换策略

后台需支持以下操作：

- 为某个虚拟模型强制指定当前主渠道
- 临时禁用某个渠道
- 暂停自动故障切换
- 恢复自动模式

### 9.5 自动故障切换策略

建议流程：

1. 主渠道请求失败
2. 判断失败类型是否可重试、可切换
3. 更新熔断计数
4. 若策略允许故障切换，则选择下一候选渠道
5. 若后续渠道成功，则返回成功响应，并记录“已切换”
6. 若全部失败，则返回统一错误

可切换失败类型建议：

- 上游超时
- 连接失败
- 429 限流
- 5xx 服务错误
- 协议可识别的临时不可用错误

## 10. 日志、审计、安全策略

### 10.1 日志策略

日志分三类：

1. 应用日志
2. 请求日志
3. 审计日志

### 10.2 脱敏策略

默认必须脱敏，尤其是：

- 对外 API Key
- 上游 API Key
- 请求正文中的敏感业务内容
- 响应中的敏感数据
- Authorization Header
- Cookie、Session

建议脱敏规则：

- Key 只保留前 6 位和后 4 位
- 文本内容只记录摘要、长度、token 估算
- Headers 使用白名单记录，不在白名单内一律隐藏
- 调试模式下也不允许打印完整密钥

### 10.3 调试开关策略

调试开关必须设计成显式开关，不允许默认开启。

建议：

- 系统级调试开关
- 渠道级调试开关
- 限时自动失效，例如 30 分钟后自动关闭
- 后台开启调试需写审计日志

### 10.4 安全策略

#### 10.4.1 密钥存储

- 上游 API Key 必须加密存储，不存明文
- 对外 API Key 不存明文，存哈希
- 加密密钥通过环境变量或本地安全配置注入

#### 10.4.2 管理后台认证

第一版建议：

- 单管理员账号密码登录
- 登录态采用 JWT + Refresh Token 或服务端 Session
- 后台接口全部鉴权
- 密码使用强哈希算法，如 bcrypt、argon2

#### 10.4.3 接口安全

- 对外接口统一 API Key 校验
- 后台接口启用 CSRF 防护或严格 SameSite Cookie 策略
- 对请求体大小做限制
- 对协议字段做白名单校验
- 禁止任意 URL 转发，防 SSRF

## 14. 关键实现建议总结

### 14.1 后端核心建议

- 使用模块化单体，不急着拆服务
- 严格分离 `handler / service / repository / adapter`
- 用接口注册表管理协议适配器和上游适配器
- 用统一请求对象承接不同对外协议
- 用路由服务承接模型映射、渠道选择和切换逻辑
- 用熔断服务承接状态机，而不是散落在调用代码里

### 14.2 前端核心建议

- 后台按业务域拆分页面和状态
- 渠道、模型、路由、熔断、日志分别建模块
- 用清晰可视化状态帮助管理员快速判断问题
- 调试开关、熔断状态、强制路由操作必须显式提示风险

### 14.3 数据与运维建议

- API Key 哈希存储，上游密钥加密存储
- Redis 做热状态，数据库做持久化
- 默认脱敏日志，调试开关限时开启
- 所有配置变更都记审计

### 14.4 可扩展性建议

第一版只做：

- 模型列表
- 文本生成
- 手动切换
- 自动熔断
- 管理后台

后续再扩展：

- 流式输出
- Embedding
- 图片生成
- 工具调用
- 多管理员权限体系
- 多租户客户端隔离
- 指标监控与告警中心

## 15. 推荐落地方案结论

本项目第一版最合适的落地方案是：

- 后端采用 Go 模块化单体
- 前端采用 Vue3 + TypeScript 管理后台
- 以“统一请求对象 + 双层适配器 + 路由服务 + 熔断服务”为核心设计
- 数据层采用关系型数据库 + Redis
- 管理侧优先覆盖渠道、模型、路由、熔断、日志、审计
- 安全侧默认开启脱敏、密钥保护、审计追踪
- 部署侧先做本地单机版，架构上预留后续拆分能力

这套方案的重点不是功能堆砌，而是把协议差异、上游差异、路由逻辑、故障处理拆干净。只要这几个边界立住，后面再接更多协议、更多渠道、更多模型，不会把代码搞成一锅烂粥。
