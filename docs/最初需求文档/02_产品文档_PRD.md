## 2. 项目背景

企业或个人团队在接入大模型能力时，通常会同时面临以下问题：

1. 上游模型来源复杂，可能同时包含官方 API 与第三方中转站，协议不统一，接入成本高。
2. 上游服务稳定性波动较大，单点故障会直接影响业务可用性。
3. 不同模型厂商的参数、返回结构、错误码差异明显，客户端适配成本高。
4. 缺少统一的密钥管理、调用审计、路由切换与故障处理能力。
5. 调试时需要可观测性，但又不能长期暴露敏感数据，存在安全与合规风险。

因此，需要建设一个“本地 AI 中转网关与管理后台”系统，作为统一接入层与运维控制台，对外屏蔽多上游差异，优先保障稳定中转能力，并支持人工干预与自动熔断。

## 3. 产品定位

### 3.1 产品定义

本产品是一个部署在本地或私有环境中的 AI 中转网关系统，向下兼容多家主流大模型协议，向上对接官方或第三方模型服务，通过统一模型名、统一鉴权、统一路由与统一日志能力，为内部业务系统提供稳定、可控、可观测的大模型接入基础设施。

### 3.2 产品定位一句话

一个以“稳定中转优先”为核心目标的本地 AI 网关与单管理员运维后台。

### 3.3 核心定位原则

- 稳定优先：优先保证调用成功率与可恢复性，而不是追求参数完全覆盖。
- 统一优先：对外暴露统一虚拟模型名，尽量减少客户端适配成本。
- 人工可控：管理员可手动切换模型路由，必要时快速止损。
- 安全默认：日志默认脱敏，调试能力默认关闭。
- MVP 克制：第一版聚焦文本生成与模型列表，不做过多计费、团队协作、复杂策略。

## 4. 产品目标

### 4.1 业务目标

- 让业务侧通过单一网关地址即可接入 OpenAI、Anthropic、Gemini 风格接口。
- 通过统一虚拟模型名与全局模型路由，降低客户端换模型、换上游的改造成本。
- 当上游异常时，支持自动熔断与人工切换，提升整体可用性。
- 为管理员提供最小可用的管理后台，完成模型路由、上游配置、客户 API Key 发放、日志查看与调试控制。

### 4.2 产品目标指标

MVP 阶段建议目标如下：

- 网关可用性：>= 99.9%（不含上游整体不可用场景）
- 路由切换生效时间：<= 10 秒
- 自动熔断识别与切换时间：<= 30 秒
- 模型列表接口成功率：>= 99.99%
- 文本生成接口可观测率：100% 记录请求元信息
- 默认敏感数据泄露风险：日志正文默认不落盘

## 5. 用户角色

当前版本仅覆盖管理后台，不覆盖客户端门户。

### 5.1 主要角色

#### 单管理员

系统唯一后台操作者，负责：

- 配置上游渠道
- 配置虚拟模型与路由关系
- 手动切换路由
- 查看日志与故障状态
- 开启、关闭临时调试
- 手动生成与禁用客户 API Key

### 5.2 非后台角色说明

以下角色不在本期后台中提供独立界面，但需作为需求边界考虑：

- 调用方系统：通过 API 调用网关
- 客户、业务方：使用管理员发放的 API Key 接入网关

## 6. 用户价值

### 6.1 对管理员的价值

- 用一个后台统一管理多个上游渠道，不用来回改配置文件。
- 遇到上游故障时能快速切换，少扯皮，能止血。
- 通过默认脱敏日志和临时调试开关，在安全和排障之间取得平衡。
- 手工发放 API Key，适合早期客户量不大、以稳为先的运营模式。

### 6.2 对调用方系统的价值

- 只接一个网关地址，降低多协议接入成本。
- 使用统一虚拟模型名，避免因上游变化频繁改代码。
- 同名优先切换，不强行大幅降级，尽量保证能力一致性。
- 核心参数统一，特殊参数透传，兼顾标准化和灵活性。

### 6.3 对业务整体的价值

- 把模型接入从“散乱直连”变成“统一控制面 + 数据面”。
- 将稳定性治理前置到网关层，减少业务系统自行兜底的复杂度。
- 为后续计费、配额、租户、多管理员等能力预留架构基础。

## 7. 使用场景

### 7.1 日常接入场景

业务系统需要调用文本生成能力，但不希望关心具体接的是哪家上游。业务只配置一个 Base URL 和一个 API Key，通过统一虚拟模型名完成调用。

### 7.2 上游故障切换场景

某个官方模型接口超时率突增，管理员在后台看到异常告警或失败率上升后，可将虚拟模型的路由切到另一个同名或兼容能力更接近的上游模型。

### 7.3 自动熔断场景

某条上游路由在一定时间窗口内连续超时或 5xx 异常达到阈值，系统自动将其熔断并切换到候选路由，同时保留事件记录供管理员追查。

### 7.4 多协议兼容场景

部分调用方使用 OpenAI SDK，部分使用 Anthropic SDK，还有部分使用 Gemini 风格接口。网关对外同时兼容多协议入口，但内部映射到统一路由配置和统一模型体系。

### 7.5 调试排障场景

客户反馈某个调用结果异常，管理员可在有限时间内开启临时调试开关，查看更详细的请求、响应片段及上游返回信息。调试结束后自动关闭，避免长期暴露敏感数据。

### 7.6 API Key 管理场景

新客户接入时，管理员手动创建客户 API Key，标记用途、状态、备注。客户异常滥用时，管理员可立即禁用对应 Key。

## 9. 产品原则

### 9.1 稳定中转优先

出现冲突时，优先保证请求可转发、可切换、可恢复，而不是优先满足所有上游私有能力。

### 9.2 同名优先切换

当虚拟模型对应上游异常时，优先切换到相同或最接近的模型名，不强行跨代大幅降级，避免输出质量和能力断崖式变化。

### 9.3 核心参数统一 + 特殊参数透传

统一处理最常用核心参数，降低客户端适配成本；对上游专有参数保留透传通道，避免把产品做死。

### 9.4 默认安全

日志默认脱敏，调试必须显式开启且带过期时间，避免因为排障方便把安全底线打穿。

### 9.5 管理后台优先服务运维决策

后台页面重点不是“好看”，而是让管理员在 30 秒内知道现在谁挂了、该切哪条路、影响了谁。

## 10. 需求总览

系统由两部分组成：

- 数据面：本地 AI 中转网关
- 控制面：管理后台

### 10.1 数据面职责

- 接收外部 API 请求
- 鉴权客户 API Key
- 识别协议类型
- 标准化核心参数
- 根据虚拟模型名匹配路由
- 转发到目标上游
- 处理异常、熔断、降级与响应转换
- 记录默认脱敏日志

### 10.2 控制面职责

- 维护上游渠道、虚拟模型、路由映射
- 查看运行状态和错误趋势
- 执行手动切换
- 管理 API Key
- 控制调试开关
- 查看审计日志和操作记录

## 12. 功能设计

### 12.1 登录

#### 12.1.1 目标

保证只有唯一管理员可进入后台。

#### 12.1.2 功能点

- 用户名、密码登录
- 登录状态维持
- 退出登录
- 登录失败提示
- 可选图形验证码或简单限流机制

#### 12.1.3 规则

- 当前版本为单管理员账号
- 连续登录失败达到阈值时，触发短暂锁定
- 登录相关日志需保留审计记录

### 12.2 首页概览

#### 12.2.1 目标

让管理员第一眼看到系统是否稳定，哪个上游有问题，最近有没有自动切换。

#### 12.2.2 页面模块

- 核心状态卡片
- 上游健康概览
- 最近 24 小时请求趋势
- 错误率趋势
- 最近熔断事件
- 最近手动切换记录
- API Key 调用 Top 列表

### 12.3 上游管理

#### 12.3.1 目标

统一管理所有可用上游，无论其为官方 API 还是第三方中转站。

#### 12.3.2 功能点

- 新增上游
- 编辑上游
- 启用、禁用上游
- 测试连通性
- 查看支持协议
- 查看上游健康状态
- 查看关联虚拟模型数量

#### 12.3.3 字段设计建议

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| 上游ID | 系统生成 | 是 | 唯一标识 |
| 上游名称 | 字符串 | 是 | 后台展示名称 |
| 上游类型 | 枚举 | 是 | `openai`、`anthropic`、`gemini` |
| 服务商类型 | 枚举 | 是 | `official`、`proxy` |
| Base URL | 字符串 | 是 | 上游基础地址 |
| API Key/Token | 密文 | 是 | 上游凭证，加密存储 |
| 默认超时(ms) | 数值 | 是 | 请求超时阈值 |
| 重试次数 | 数值 | 是 | 单次调用重试次数 |
| 启用状态 | 布尔 | 是 | 是否可参与路由 |
| 健康检查开关 | 布尔 | 是 | 是否纳入健康探测 |
| 备注 | 字符串 | 否 | 运营说明 |

### 12.4 虚拟模型管理

#### 12.4.1 目标

对外提供统一虚拟模型名，屏蔽上游真实模型差异。

#### 12.4.2 功能点

- 新增虚拟模型
- 编辑虚拟模型
- 配置协议兼容类型
- 绑定候选上游模型
- 设置首选模型和备选模型
- 查看当前路由状态

### 12.5 路由管理

#### 12.5.1 目标

让管理员直观看到每个虚拟模型当前实际走到哪个上游，并可快速切换。

#### 12.5.2 核心能力

- 查看虚拟模型当前生效路由
- 手动切换到候选上游
- 查看自动熔断状态
- 查看切换原因
- 查看切换历史

### 12.6 自动熔断

#### 12.6.1 目标

在管理员不在线时，系统也能自动止损，避免上游持续报错拖垮整体成功率。

#### 12.6.2 触发条件建议

- 连续失败次数阈值
- 时间窗口内失败率阈值
- 时间窗口内超时率阈值
- 连续 5xx 次数阈值
- 健康检查连续失败次数

### 12.7 API Key 管理

#### 12.7.1 目标

由管理员手动发放客户 API Key，控制调用入口。

#### 12.7.2 功能点

- 创建 Key
- 查看 Key 列表
- 复制一次性明文 Key
- 启用、禁用 Key
- 查看最近使用时间
- 查看调用量概览
- 设置备注

### 12.8 日志中心

#### 12.8.1 目标

在默认安全前提下，提供足够的问题排查能力。

#### 12.8.2 日志分类

- 请求日志
- 错误日志
- 路由切换日志
- 熔断事件日志
- 管理员操作审计日志
- 调试日志

### 12.9 调试控制

#### 12.9.1 目标

允许管理员在必要时临时提升日志详细度，但必须有明确边界。

#### 12.9.2 功能点

- 开启临时调试
- 设置调试有效时长
- 选择生效范围
- 查看剩余时间
- 手动提前关闭

### 12.10 系统设置

#### 12.10.1 功能点

- 修改管理员密码
- 熔断阈值配置
- 默认超时配置
- 日志保留策略
- 系统基础信息查看

## 13. 对外接口能力设计

### 13.1 核心接口范围

MVP 仅支持：

- 文本生成
- 模型列表

### 13.2 协议兼容目标

#### OpenAI 风格

建议兼容：

- `POST /v1/chat/completions`
- `GET /v1/models`

#### Anthropic 风格

建议兼容：

- `POST /v1/messages`
- `GET /v1/models`

#### Gemini 风格

建议兼容文本生成与模型查询的 Gemini 风格入口，具体路径可按实现方案映射，但管理后台中应统一为文本生成与模型列表两类能力。

### 13.3 参数策略

建议统一以下核心参数语义：

- `model`
- `messages` / 输入内容
- `temperature`
- `top_p`
- `max_tokens`
- `stream`
- `stop`

以下原则适用于厂商专有参数：

- 网关不强行解释所有特殊参数
- 若当前上游支持，则可原样透传
- 若当前上游不支持，可忽略或返回提示性错误
- 核心参数优先使用统一字段

## 15. 关键交互设计

### 15.1 新增上游

1. 管理员进入上游管理页面。
2. 点击“新增上游”。
3. 填写名称、类型、Base URL、凭证、超时等信息。
4. 点击“测试连接”。
5. 测试成功后可保存；测试失败允许保存但需有明确风险提示。
6. 保存后进入列表，默认展示健康状态。

### 15.2 新增虚拟模型并绑定候选

1. 进入虚拟模型页面。
2. 创建虚拟模型，填写对外名称和主协议。
3. 选择一个或多个上游候选模型。
4. 设置优先级、是否同名。
5. 保存后自动生成路由记录。

### 15.3 手动切换路由

1. 管理员在路由列表看到某模型异常。
2. 打开路由详情。
3. 查看候选目标健康状态。
4. 选择目标并点击“切换”。
5. 可选设置手动锁定时长。
6. 确认后即时生效，并写入切换记录。

### 15.4 开启临时调试

1. 管理员进入调试控制。
2. 选择范围（全局、模型、API Key）。
3. 选择时长。
4. 点击开启。
5. 页面顶部显示调试中 Banner 和剩余时间。
6. 到期自动关闭。

### 15.5 创建 API Key

1. 管理员点击“创建 Key”。
2. 填写客户名称、备注。
3. 点击确认生成。
4. 弹窗显示完整 Key，一次性复制。
5. 关闭后仅可见前缀、尾号。

## 17. 异常场景设计

### 17.1 上游超时

- 记录超时错误日志
- 命中统计计数
- 达到阈值则触发熔断
- 若存在备选路由则自动切换
- 返回统一错误结构给调用方

### 17.2 上游鉴权失败

- 立即记录高优先级错误
- 标记该上游健康状态异常
- 不建议自动重试多次
- 若可切换则自动切换

### 17.3 客户 API Key 无效

- 直接拦截，不转发上游
- 返回统一鉴权失败错误
- 记录安全日志

### 17.4 无可用候选路由

- 立即标记该虚拟模型不可服务
- 返回明确错误信息
- 生成高优先级事件记录

### 17.5 参数不兼容

- 核心参数继续正常处理
- 特殊参数按策略忽略或提示
- 调试模式下记录详细冲突信息

### 17.6 日志敏感信息风险

- 依然保留强脱敏规则
- 超时自动关闭调试
- 记录调试开启与关闭事件
- 调试日志缩短保留周期

### 17.7 管理员误操作切换

- 切换前二次确认
- 保留切换历史
- 支持快速回切
- 支持手动锁定与解除

## 19. 非功能需求

### 19.1 性能要求

- 模型列表接口应轻量返回，支持快速缓存
- 文本生成链路不引入明显额外延迟
- 后台列表页支持分页和基础筛选

### 19.2 稳定性要求

- 路由变更需热更新生效，避免重启服务
- 熔断状态应具备持久化能力
- 上游异常不得影响后台登录与配置管理

### 19.3 安全要求

- 后台鉴权安全可靠
- 上游凭证加密存储
- API Key 不明文长期保存
- 默认脱敏日志
- 调试模式自动过期

### 19.4 可维护性要求

- 核心对象命名统一
- 前后端字段语义一致
- 路由切换、熔断、调试控制等关键流程要有清晰状态机

## 23. 风险与注意事项

### 23.1 协议兼容复杂度风险

虽然对外支持 OpenAI、Anthropic、Gemini，但不同协议在参数结构、错误格式、流式返回上差异较大。MVP 必须先收敛在文本生成和模型列表，别一上来想全兼容。

### 23.2 第三方中转站不稳定风险

第三方中转站的稳定性、错误码规范性、限流策略可能很野。产品与技术需要把“异常归一化”和“快速切换”作为优先级最高能力。

### 23.3 模型名映射风险

统一虚拟模型名会让客户端更爽，但如果映射策略不透明，可能造成结果预期偏差。因此后台需要清楚展示当前虚拟模型实际落到哪里，以及切换历史。

### 23.4 调试日志安全风险

调试能力是必要的，但绝不能默认长期打开。必须配强提醒、自动关闭和审计记录。

## 24. 结论

本项目 V1 的核心不是做成一个“大而全的 AI 平台”，而是做成一个“本地可控、稳定优先、出事能快速止血”的 AI 中转网关。

产品上必须死守三条线：

- 对外统一：统一协议入口、统一虚拟模型名、统一鉴权方式
- 对内可控：上游可管理、路由可切换、异常可熔断
- 默认安全：日志脱敏、调试临时、审计完整
