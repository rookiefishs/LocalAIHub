## 2. 设计目标

本接口文档用于明确第一版系统的对外网关接口和后台管理接口，重点解决以下问题：

- 对外如何兼容 OpenAI、Anthropic、Gemini 三类协议
- 内部如何统一错误结构、请求标识、鉴权方式
- 后台管理接口如何覆盖上游、虚拟模型、路由、日志、调试开关等核心能力
- 开发、联调、测试各方如何基于统一字段和约定推进实现

第一版只覆盖：

- 文本生成接口
- 模型列表接口
- 管理后台核心配置接口

第一版不覆盖：

- 图片、音频、Embedding、文件上传
- 客户门户接口
- 计费、对账、套餐接口

## 3. 接口设计总原则

### 3.1 总体原则

- 对外网关接口尽量兼容主流协议习惯
- 管理后台接口统一走内部 REST 风格
- 每个请求都生成或透传 `X-Request-Id`
- 错误返回必须统一，不把上游原始脏错误直接甩给客户端
- 响应体字段命名尽量稳定，不要今天 snake_case 明天 camelCase

### 3.2 鉴权原则

#### 3.2.1 网关接口鉴权

- 使用客户端 API Key 进行鉴权
- 推荐 Header：`Authorization: Bearer <client_api_key>`
- 同时兼容部分协议默认 Header 格式时，内部统一提取 Bearer Token

#### 3.2.2 后台接口鉴权

- 使用管理员登录态鉴权
- 推荐使用 `HttpOnly Cookie + Session/JWT` 方案
- 后台接口前缀统一为 `/admin/api/v1`

### 3.3 请求 ID 约定

- 所有接口响应 Header 返回 `X-Request-Id`
- 若客户端传入 `X-Request-Id`，服务端可选择保留或重写
- 日志、审计、熔断、错误排查统一依赖该字段串联

### 3.4 时间与时区约定

- 所有时间字段统一使用 ISO 8601 格式
- 后台接口默认返回 UTC 或带时区偏移的时间字符串
- 数据库存储建议统一 UTC

### 3.5 分页约定

后台列表接口统一采用：

- `page`
- `page_size`

统一返回：

- `items`
- `total`
- `page`
- `page_size`

## 5. 通用响应结构

### 5.1 后台接口统一响应

#### 成功响应

```json
{
  "code": 0,
  "message": "ok",
  "data": {},
  "request_id": "req_01HXYZ..."
}
```

#### 失败响应

```json
{
  "code": 400100,
  "message": "invalid parameters",
  "data": null,
  "request_id": "req_01HXYZ..."
}
```

### 5.2 分页响应结构

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "items": [],
    "total": 120,
    "page": 1,
    "page_size": 20
  },
  "request_id": "req_01HXYZ..."
}
```

### 5.3 网关错误响应统一约定

虽然对外兼容多协议，但内部统一错误分类如下：

- `unauthorized`
- `forbidden`
- `invalid_request`
- `model_not_found`
- `model_unavailable`
- `upstream_timeout`
- `upstream_rate_limited`
- `upstream_auth_failed`
- `upstream_internal_error`
- `internal_error`

建议不同协议对外做本协议风格包装，但内部错误码保持统一映射。

## 7. 对外网关接口设计

## 7.1 OpenAI 风格文本接口

### 7.1.1 接口信息

- 方法：`POST`
- 路径：`/proxy/openai/v1/chat/completions`
- 鉴权：`Authorization: Bearer <client_api_key>`

### 7.1.2 请求体示例

```json
{
  "model": "gpt-4o",
  "messages": [
    {
      "role": "system",
      "content": "You are a helpful assistant."
    },
    {
      "role": "user",
      "content": "你好，帮我写一段 Go 路由中间件示例"
    }
  ],
  "temperature": 0.7,
  "top_p": 0.9,
  "max_tokens": 512,
  "stream": false
}
```

### 7.1.3 字段说明

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| model | string | 是 | 虚拟模型名 |
| messages | array | 是 | 对话消息数组 |
| temperature | number | 否 | 核心采样参数 |
| top_p | number | 否 | 核心采样参数 |
| max_tokens | integer | 否 | 最大输出 token |
| stream | boolean | 否 | 是否流式返回 |
| stop | array/string | 否 | 停止词 |
| 其他字段 | mixed | 否 | 特殊参数透传候选 |

### 7.1.4 成功响应示例

```json
{
  "id": "chatcmpl_local_001",
  "object": "chat.completion",
  "created": 1774800000,
  "model": "gpt-4o",
  "choices": [
    {
      "index": 0,
      "message": {
        "role": "assistant",
        "content": "下面给你一个基于 Gin 的 Go 路由中间件示例。"
      },
      "finish_reason": "stop"
    }
  ],
  "usage": {
    "prompt_tokens": 32,
    "completion_tokens": 66,
    "total_tokens": 98
  }
}
```

### 7.1.5 失败响应示例

```json
{
  "error": {
    "message": "virtual model not found",
    "type": "invalid_request_error",
    "code": "GW404001"
  }
}
```

## 8. 对外参数统一与透传规则

### 8.1 核心统一字段

内部统一对象至少包含：

- `model`
- `messages` / `contents` / `prompt`
- `temperature`
- `top_p`
- `max_tokens`
- `stream`
- `stop`

### 8.2 透传字段规则

- 透传字段必须经过适配器白名单筛选
- 核心字段与透传字段冲突时，以核心字段为准
- 不允许客户端透传影响内部路由控制的字段
- 不允许透传会泄露上游认证信息的字段

### 8.3 参数不兼容处理

- 可忽略时：记录调试信息，不中断主流程
- 不可忽略时：返回 `GW422001`
- 后台日志记录具体适配失败原因

## 21. 安全约束

- 后台创建客户 API Key 后仅展示一次完整明文
- 后台任何查询接口不返回上游完整 key
- 日志详情接口默认只展示脱敏摘要
- 调试会话接口必须记录审计日志
- 任何资源删除类接口第一版建议优先做逻辑删除或禁用，不做硬删除
