## 2. 项目背景

当前 AI 模型生态高度碎片化，不同上游厂商和中转站在以下方面存在明显差异：

- 协议不统一：OpenAI、Anthropic、Gemini 及各类中转站接口规范不同
- 模型命名不统一：同能力模型在不同上游下名称不同，客户端接入成本高
- 稳定性不可控：单一上游波动、限流、封禁、错误率飙升时，业务容易中断
- 密钥管理复杂：存在单 key、key 池、不同上游多账号混用等情况
- 问题排查困难：日志不能全量保留明文，但出问题时又需要临时定位

因此需要建设一套“本地 AI 中转网关与管理后台”，作为统一接入层和上游路由层，对客户端暴露统一协议和统一虚拟模型名，对内部实现多上游接入、模型映射、故障切换、日志审计与基础运维管理，优先保证“稳定中转”。

该项目第一阶段以“本地开发版”落地，重点解决可用性、统一性、可观测性和切换能力，不追求复杂权限体系、多租户、计费、正式限流等平台化能力。

## 3. 项目目标

### 3.1 核心目标

本项目的首要目标是：**稳定中转优先**。

围绕该目标，第一版系统需实现以下能力：

1. 对客户端提供统一 AI 调用入口，降低接入复杂度
2. 支持多协议对外兼容，适配不同客户端或 SDK
3. 将多个上游模型映射为统一虚拟模型名，屏蔽真实上游差异
4. 支持管理员手动切换模型路由，并在上游异常时自动熔断切换
5. 在单 key 和多 key 模式下均可运行
6. 默认日志脱敏，同时支持临时调试开关，兼顾安全与排障
7. 提供单管理员后台，完成上游配置、模型映射、路由切换、日志查看等管理操作

### 3.2 非目标

以下内容不属于第一版核心目标：

- 不做正式限流系统
- 不做复杂 RBAC 权限体系
- 不做多管理员、多角色、多租户
- 不做在线自助开通与用户注册
- 不做计费、配额、账单与财务对账
- 不做图像、音频、文件、向量库等扩展能力
- 不做分布式部署、高可用集群、云端托管版
- 不做细粒度按用户、按 Key、按路由规则的复杂调度

## 4. 项目范围

### 4.1 范围内

第一版范围明确包含：

#### 4.1.1 网关能力

- 提供统一 API 入口
- 支持多协议同时对外
- 支持文本生成类接口
- 支持模型列表接口
- 支持 API Key 校验
- 支持统一虚拟模型名访问
- 支持核心参数统一转换
- 支持特殊参数透传到上游
- 支持全局模型路由
- 支持同名模型优先降级切换
- 支持手动切换与自动熔断

#### 4.1.2 上游接入能力

- 支持 OpenAI
- 支持 Anthropic
- 支持 Gemini
- 支持各类兼容中转站
- 支持单 key 模式
- 支持多 key 模式
- 支持按上游维护可用状态

#### 4.1.3 管理后台能力

- 单管理员登录后台
- 配置上游渠道及密钥
- 配置虚拟模型与真实模型映射
- 配置默认路由与候选降级链路
- 查看请求日志与错误日志
- 临时开启调试日志
- 手动启停上游、切换路由
- 查看基础运行状态

### 4.2 范围外

第一版明确不做：

- 用户后台、自助申请 Key、门户站点
- 用户级用量统计与分析报表
- 套餐、订单、支付
- 企业多租户隔离
- WebSocket 流式协议扩展以外的复杂通信协议
- 正式 WAF、网关集群、分布式限流
- 自动化告警通知集成可列为后续项

## 5. 用户角色

### 5.1 外部客户端

指接入该网关的业务系统、脚本、应用服务、前端代理层或开发者工具。

特点：

- 只通过 API Key 完成身份校验
- 不关心真实上游是谁
- 只面向统一虚拟模型名发起调用
- 期望接口稳定、返回格式统一、错误可理解

主要诉求：

- 接入简单
- 尽量不改现有调用方式
- 模型名称稳定
- 上游切换时无感或少感知

### 5.2 单管理员

指系统唯一后台管理者，负责本地版的全部管理和维护工作。

职责：

- 配置上游连接与密钥
- 配置虚拟模型与真实模型映射
- 维护路由策略
- 在异常时手动切换
- 开启或关闭调试模式
- 查询日志、定位问题

权限边界：

- 拥有全部后台管理权限
- 不需要复杂权限拆分
- 不提供次级管理员和审计审批流

## 6. 产品定位与核心原则

### 6.1 产品定位

本系统不是通用 AI 平台，也不是面向最终用户的聊天产品，而是一个：

- 面向开发接入的 AI 中转网关
- 面向运维配置的管理后台
- 面向稳定性的路由与故障切换层

### 6.2 核心原则

#### 6.2.1 稳定优先

任何设计优先保证请求不断、尽量可切换、尽量少人工干预。

#### 6.2.2 统一优先

客户端看到的是统一模型名、统一参数结构、统一错误风格，不暴露上游碎片化差异。

#### 6.2.3 安全优先

日志默认脱敏，敏感信息不明文落盘；调试能力必须可控、可关闭、可追溯。

#### 6.2.4 简化优先

第一版只做必要能力，不堆复杂规则，不过度设计，先把主链路跑稳。

## 7. 业务场景说明

### 7.1 统一接入场景

业务系统原本需要分别对接 OpenAI、Anthropic、Gemini 或某个中转站。接入本系统后，只需要调用本地网关地址并使用平台分发的 API Key，通过统一模型名访问即可。

价值：

- 降低客户端接入改造成本
- 屏蔽上游差异
- 后续替换上游不影响客户端

### 7.2 上游故障切换场景

当某虚拟模型当前绑定的主上游出现超时、返回错误、连接失败、认证失败或持续异常时，系统可按既定策略自动熔断，并切换到候选上游。

价值：

- 减少服务中断
- 缩短人工响应时间
- 提高整体可用性

### 7.3 临时问题排查场景

默认情况下，请求与响应日志均脱敏保存。当某上游返回异常或参数不兼容时，管理员可短时开启调试日志，对指定时间窗口内的请求记录更详细内容，用于定位问题，排查后再关闭。

价值：

- 平时更安全
- 出问题时能查
- 避免长期明文日志风险

### 7.4 模型平滑替换场景

客户端长期使用统一虚拟模型名，例如 `gpt-4o`、`claude-sonnet`、`gemini-pro` 等虚拟名。管理员可在后台将其实际映射从某个中转站切换到官方上游或另一个兼容上游，而无需通知客户端修改模型名。

价值：

- 降低切换成本
- 便于灰度和迁移
- 减少业务方感知

## 8. 核心业务流程

### 8.1 客户端调用文本接口流程

1. 客户端携带 API Key 调用网关文本接口
2. 网关校验 API Key 是否有效
3. 网关解析请求协议与请求体
4. 识别客户端请求的虚拟模型名
5. 根据全局模型路由配置查找当前主上游
6. 进行核心参数标准化转换
7. 将特殊参数按透传规则带给目标上游
8. 发送请求到对应上游
9. 接收上游响应并转换为统一对外格式
10. 记录脱敏日志并返回客户端

### 8.2 自动熔断与降级流程

1. 某上游请求出现连续失败、超时或触发熔断条件
2. 系统将该上游该模型路由标记为异常状态
3. 若存在同名虚拟模型的候选上游，按优先级切换
4. 后续请求优先走新的可用上游
5. 后台展示当前熔断状态、触发原因和切换结果
6. 管理员可手动恢复、手动切回或继续观察

### 8.3 管理员配置上游流程

1. 管理员进入后台上游管理页
2. 新增一个上游渠道
3. 选择上游类型：OpenAI、Anthropic、Gemini、中转站
4. 填写基础信息：名称、Base URL、认证方式、key 模式等
5. 录入单 key 或多 key
6. 保存并进行连通性测试
7. 测试通过后启用该上游

### 8.4 管理员配置模型映射流程

1. 管理员进入模型管理页
2. 新增或编辑虚拟模型名
3. 配置该虚拟模型可映射的真实上游模型
4. 设置主路由与候选路由顺序
5. 保存后立即生效或按配置热更新生效
6. 后续客户端统一调用该虚拟模型名

### 8.5 调试日志流程

1. 管理员在后台打开临时调试开关
2. 系统记录开启人、开启时间、过期时间、作用范围
3. 在生效期内记录增强日志
4. 到期自动关闭，或管理员手动关闭
5. 日志仍需做部分敏感字段保护，不允许无限制明文泄露

## 9. 功能清单

### 9.1 网关对外接口模块

#### 9.1.1 文本接口

系统需提供文本生成类接口，至少满足以下要求：

- 支持统一文本请求入口
- 支持同步返回模式
- 支持流式返回能力作为优先兼容项
- 支持常见消息结构输入
- 支持系统消息、用户消息、助手消息等基本角色
- 支持统一模型字段输入
- 返回统一格式的结果对象
- 能将不同上游错误转换为统一错误结构

重点说明：

- 第一版接口范围只覆盖文本接口，不扩展图片、音频、Embedding 等
- 不同协议下文本接口可做兼容适配，但内部统一走一套标准请求对象

#### 9.1.2 模型列表接口

系统需提供模型列表接口，至少满足：

- 返回当前可供客户端调用的虚拟模型列表
- 不直接暴露全部真实上游模型
- 支持模型状态标识，如可用、部分可用、维护中
- 可返回描述信息，如模型说明、默认能力标签
- 可根据后台配置实时变化

### 9.2 协议适配模块

由于确认需求为“多协议同时做”，系统需要支持多个对外协议风格。

第一版要求：

- 支持 OpenAI 风格接口适配
- 预留 Anthropic、Gemini 风格对外协议适配能力
- 后端内部统一转换为标准请求结构
- 响应也需从内部标准结构转换回对应协议格式
- 协议差异点需在参数映射层处理，而不是散落在业务逻辑层

适配原则：

- 对客户端尽量兼容现有主流 SDK 的调用习惯
- 不盲目追求全量协议细节覆盖，第一版先覆盖文本与模型列表
- 遇到不兼容参数时，遵循“核心参数统一 + 特殊参数透传”的策略

### 9.3 API Key 校验模块

系统需通过 API Key 对客户端完成基础认证。

要求：

- 所有对外接口均需校验 API Key
- API Key 为当前唯一客户身份识别方式
- API Key 校验失败时返回统一鉴权错误
- API Key 支持手动生成、手动分发、手动停用
- API Key 需支持备注字段，便于识别用途或客户
- 第一版无需用户注册、无需 OAuth、无需账号密码体系

建议补充：

- API Key 数据库存储时建议只保存摘要或加密值
- 后台仅在创建时展示完整 Key，后续不可见或仅显示部分掩码

### 9.4 上游管理模块

系统需支持多种上游类型的统一管理。

#### 9.4.1 支持的上游类型

- OpenAI 官方或兼容服务
- Anthropic 官方或兼容服务
- Gemini 官方或兼容服务
- 各类第三方中转站

#### 9.4.2 上游配置字段

每个上游至少包括：

- 上游名称
- 上游类型
- Base URL
- 协议类型
- 鉴权方式
- key 模式：单 key、多 key
- 超时配置
- 重试配置
- 启用状态
- 备注信息

#### 9.4.3 key 管理要求

- 支持单 key 配置
- 支持多 key 配置
- 多 key 模式下可维护 key 列表状态
- 支持 key 启用、停用、失效标记
- key 使用失败时可按策略剔除或降权
- 第一版不强制实现复杂负载均衡，但至少要支持基础可用切换

### 9.5 虚拟模型管理模块

这是系统的核心功能之一。

要求：

- 系统对外只暴露虚拟模型名
- 每个虚拟模型可关联一个或多个真实上游模型
- 同一个虚拟模型可在不同上游下维护不同映射
- 管理员可配置主路由和候选路由
- 支持配置模型是否对外可见
- 支持配置模型说明、用途标签、排序

### 9.6 路由管理模块

根据已确认需求，第一版采用**全局模型路由**，不做按客户、按地域、按业务线的细粒度路由。

要求：

- 每个虚拟模型维护一套全局生效路由
- 路由至少支持主上游和候选上游顺序
- 支持管理员手动切换当前主上游
- 支持系统自动熔断后切换
- 支持展示当前路由状态和最近切换记录
- 支持同名优先切换

### 9.7 熔断与故障切换模块

系统需支持自动熔断，核心是防止异常上游持续拖垮请求成功率。

第一版建议至少支持以下可配置条件：

- 连续请求失败达到阈值
- 指定时间窗口内失败率超过阈值
- 超时次数达到阈值
- 认证错误持续发生
- 上游连接失败持续发生

### 9.8 参数标准化与透传模块

根据需求，系统需采用“核心参数统一 + 特殊参数透传”策略。

建议统一的核心参数包括：

- `model`
- `messages`
- `temperature`
- `top_p`
- `max_tokens`
- `stream`
- `stop`

对某些上游专有参数，系统允许透传，但需要白名单和冲突处理规则。

### 9.9 日志与调试模块

日志策略为：**默认脱敏 + 支持临时调试开关**。

建议记录：

- 请求时间
- 请求 ID
- 客户 API Key 标识摘要
- 对外协议类型
- 虚拟模型名
- 目标上游名称
- 目标真实模型名
- 请求耗时
- 响应状态
- 错误码
- token 统计
- 熔断、切换信息

### 9.10 后台管理模块

后台需支持：

- 上游管理
- API Key 管理
- 虚拟模型管理
- 路由管理
- 调试开关管理
- 系统基础设置

### 9.11 基础保护能力

虽然第一版不做正式限流，但建议提供低成本基础保护，避免系统裸奔。

建议项：

- 单 IP 基础访问频率保护
- 请求体大小限制
- 并发连接上限保护
- 管理后台登录失败次数保护
- 基础黑名单能力
- 超时控制与资源释放

## 11. 数据对象设计

## 12. 非功能需求

### 12.1 性能要求

- 单机环境下可支撑日常开发测试与小规模业务接入
- 请求处理链路开销应可控，不能因为中转层导致明显额外高延迟
- 日志写入不得阻塞主请求链路
- 熔断判断与路由查询需轻量化

### 12.2 可用性要求

- 当单个上游故障时，不应影响其他上游继续服务
- 自动熔断与切换应尽量在短时间内生效
- 后台配置变更应可热更新或低中断生效
- 系统异常时应有足够日志辅助排查

### 12.3 安全要求

- API Key、上游 key 必须加密存储或摘要存储
- 后台登录口令必须加密保存
- 日志默认脱敏
- 调试模式必须可过期
- 后台接口需有登录态校验

### 12.4 可维护性要求

- 后端模块边界清晰：协议适配、路由、熔断、日志、上游调用分层
- 前端页面按模块拆分，避免单页逻辑爆炸
- 配置项命名统一，便于后续扩展
- 核心行为如切换、熔断、调试开启应有明确操作记录

### 12.5 可扩展性要求

- 可扩展更多协议
- 可扩展更多上游类型
- 可扩展更多接口类型
- 可扩展多管理员和 RBAC
- 可扩展正式限流、配额、计费

## 13. 风险与约束

### 13.1 上游协议差异风险

OpenAI、Anthropic、Gemini 及各种中转站接口行为并不完全一致。即便表面兼容，也可能在参数名、消息结构、错误码、流式格式上存在坑。

### 13.2 同名模型能力不完全一致风险

“同名优先切换”能降低客户端感知，但不同上游对同名模型的真实能力、上下文长度、输出稳定性可能仍有差异。

### 13.3 日志调试与数据安全风险

排查问题时若开放过度日志，容易泄露用户输入、敏感业务数据或密钥。

### 13.4 单管理员风险

单管理员简化了第一版，但也带来单点操作风险。

### 13.5 无正式限流风险

第一版不做正式限流，如果 API Key 泄露、脚本误调用或恶意打流量，可能压垮本地服务或打爆上游。

## 14. 验收标准

### 14.1 功能验收

- 客户端可通过 API Key 调用文本接口
- 客户端可获取模型列表
- 后台可新增虚拟模型并完成多上游映射
- 可配置主路由和候选路由
- 管理员可手动切换路由
- 上游异常时可自动熔断切换
- 可新增 OpenAI、Anthropic、Gemini、中转站类型上游
- 支持单 key 与多 key 模式
- 默认日志脱敏
- 可开启临时调试开关并自动关闭

### 14.2 质量验收

- 正常请求链路稳定，无明显重复性错误
- 故障切换逻辑可复现并可验证
- 关键错误信息可追踪到请求 ID
- 后台主要页面操作流畅
- 无明显敏感信息明文泄露

### 14.3 验收用例建议

1. 单一上游正常调用文本接口成功
2. 多个上游下虚拟模型映射正确
3. 主上游故障后自动切换到候选上游
4. 管理员手动切换后客户端调用走新路由
5. API Key 停用后请求被拒绝
6. 默认日志不显示完整密钥与完整敏感文本
7. 调试模式开启后可看到更详细信息，过期自动关闭
8. 多 key 模式下某个 key 失效后可切换到其他 key
9. 模型列表接口只返回对外可见模型
10. 非法参数或不支持参数返回统一错误结构

## 17. 产品结论

本项目第一版的定位非常明确：**做一个本地可运行、稳定优先、统一接入、可切换、可管理、可排障的 AI 中转网关与后台**。

第一版关键不在于功能铺多，而在于把下面几件事干稳：

- 客户端接入足够简单，只认 API Key 和虚拟模型名
- 上游接入足够灵活，能兼容多种官方与中转站
- 路由能力足够清晰，支持手动切换与自动熔断
- 日志策略足够稳，平时安全，出问题能查
- 后台管理足够顺手，不需要改代码才能运维
